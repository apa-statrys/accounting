## Background

For reconciliation, the client supplies supporting documents from their transaction histories. Statrys BA transactions and other-bank-account transactions are shown in two separate histories. Reconciliation appears to the client only as an attachment dimension: each transaction is either Missing an Attachment or has an Attachment Added, and the client attaches documents where needed. The actual matching/reconciliation is performed by the system and the accountant — the client's job is just to provide documents.

## Goal

Give the client 2 or more separate transaction histories (Statrys BA and Other Bank Accounts) with an attachment-status filter and document upload, so they can supply the documents needed for matching — in plain "attachment" language, not accounting terms.

---

## Acceptance Criteria

1. **Two separate transaction histories (can be more if Client has more than 1 additional account)**

GIVEN the client opens their transactions
WHEN the histories load
THEN the Statrys BA transactions history and the Other Bank Accounts transactions history are displayed as two separate histories
AND the Other Bank Accounts history is populated from imported statements

2. **Attachment status per transaction**

GIVEN transactions are displayed in either history
THEN each transaction shows its attachment status: Missing Attachment or Attachment Added (see Attachment Status table)
AND reconciliation/matching states are not shown to the client (handled by the system and accountant)

3. **Filter, search and sort**

GIVEN a transactions history is displayed
WHEN the client filters by attachment status, searches, or sorts
THEN the list is narrowed accordingly (e.g. show only Missing Attachment)

4. **Open a transaction detail**

GIVEN the client selects a transaction
WHEN it opens
THEN its details are displayed read-only, including any attached document

5. **Attach a document to a transaction**

GIVEN a transaction with Missing Attachment
WHEN the client uses its per-transaction upload
THEN they can attach a supporting document to that transaction, moving it to Attachment Added (detailed in Client Story C3)

6. **Bulk attach documents**

GIVEN the client wants to provide several documents
WHEN they use bulk upload
THEN they can submit multiple supporting documents via email or WhatsApp (detailed in Client Story C3)

## Rules

**Transaction Histories (separate surfaces)**

| History | Source |
| --- | --- |
| Statrys BA transactions | Statrys Business Account |
| Other Bank Accounts transactions | Imported other-account statements |

**Attachment Status (client-visible)**

| Status | Meaning |
| --- | --- |
| Missing Attachment | The transaction needs a supporting document |
| Attachment Added | A supporting document has been provided |
| No attachment needed _(optional)_ | Transaction that never needs a document (e.g. bank fee, interest, internal transfer) |

**Transaction Columns**

| Column | Display Rule |
| --- | --- |
| Date | Transaction date |
| Description | Transaction description / counterparty |
| Amount | Currency abbreviation + amount |
| Attachment status | Missing Attachment / Attachment Added / No attachment needed |
| Document | Attached document (if any) |

## Notes

* Client platform only. The two histories are kept separate (Statrys BA vs Other Bank Accounts).
* Client-facing language is "attachment," not reconciliation/matched — the client only provides documents.
* Matching/reconciliation (including auto-reconcile of high-confidence) is done by the system and accountant and is not surfaced to the client.
* "No attachment needed" state exist for the client (so bank fees / interest / internal transfers don't show as Missing), or only Missing / Added?
* System detects which transactions require a document)
