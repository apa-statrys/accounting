## Background

To reconcile transactions, the client provides supporting documents for transactions that are Missing an Attachment. They can attach a document directly to a specific transaction, or send documents in bulk (in-app, email, or WhatsApp). Bulk documents go into a pool and the system OCR+AI engine matches each one to its transaction automatically. Once a document is attached, the transaction becomes Attachment Added and the reconciliation matching runs.

## Goal

Let the client supply supporting documents — per-transaction or in bulk via in-app / email / WhatsApp — with bulk documents auto-matched to transactions, so Missing Attachment transactions become Attachment Added.

---

## Acceptance Criteria

1. **Attach a document to a specific transaction**

GIVEN a transaction with Missing Attachment
WHEN the client uploads a document via its per-transaction upload
THEN the document is attached to that transaction
AND the transaction status changes to Attachment Added and reconciliation matching is triggered

2. **In-app bulk upload**

GIVEN the client is on the Missing Attachment list
WHEN they drag/drop or select multiple files
THEN the documents are submitted to the documents pool for auto-matching (see AC4)

3. **Send documents by email or WhatsApp**

GIVEN the client has supporting documents to send
WHEN they send one or multiple documents via the email forwarding address or WhatsApp
THEN the documents are received into the documents pool for auto-matching (see AC4)

4. **Bulk documents are auto-matched to transactions**

GIVEN documents are in the pool (from in-app bulk, email, or WhatsApp)
WHEN the system runs OCR+AI matching
THEN each document is matched to its transaction and that transaction becomes Attachment Added
AND a document that cannot be matched is flagged for review (see AC5)

5. **Unmatched documents are flagged**

GIVEN a pooled document that the engine cannot match to a transaction
WHEN matching completes
THEN the document is flagged as unmatched for follow-up (client prompt and/or accountant review)

6. **File rules**

GIVEN the client uploads or sends a document
THEN only accepted file types and size are allowed (see File Rules table)
AND invalid or duplicate files are rejected with an error

---

## Rules

**Upload Methods**

| Method | Behaviour |
| --- | --- |
| Per-transaction upload | Attach directly to one transaction (immediate Attachment Added) |
| In-app bulk upload | Multiple files from the Missing Attachment list → pool |
| Email | Single/multiple documents to the forwarding address → pool |
| WhatsApp | Documents sent via WhatsApp → pool |

**File Rules**

| Rule | Value |
| --- | --- |
| Accepted formats | PDF, JPG, PNG |
| Max size | 25 MB (To be confirmed by Tech Team) |
| Duplicate / invalid | Rejected with an error |

**Matching**

| Source | Matching |
| --- | --- |
| Per-transaction | Attached to that transaction directly |
| Bulk (in-app / email / WhatsApp) | Pooled, then OCR+AI auto-matches each to its transaction; unmatched documents flagged |

---

## Notes

* Client platform only. The OCR+AI matching is the same reconciliation engine.
* Attaching a document moves the transaction to Attachment Added and triggers reconciliation matching.
* Email and WhatsApp intake rely on the channels configured in the deferred Docs Collection / settings block.
* An unmatchable pooled document sits for the accountant
* Single transaction have multiple attachments
